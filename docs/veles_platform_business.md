# Клуб по подписке «Велес»: бизнес-логика и архитектура платформы v6

Дата фиксации: 2026-06-16

Документ фиксирует актуальную бизнес-логику и архитектурную модель подписочной платформы клуба «Велес».

Основной фокус проекта: управляемый доступ в новые закрытые Telegram-ресурсы через собственный backend, личный кабинет, единую базу пользователей, подписки, платежи и партнёрскую программу.

Примечание: документ исторически содержит и бизнес-логику, и архитектурные подробности. Сейчас они не переписываются с нуля; позже технические детали нужно вынести в отдельный архитектурный документ или ADR после approval владельца.

---

## 1. Главный принцип архитектуры

Центральный backend является единственным источником правды.

Telegram-боты, сайт, личный кабинет, новые закрытые чаты, старые чаты и VK-сообщество не принимают самостоятельных финальных решений по пользователям, подпискам, доступам, оплатам, реферальной программе и партнёрским начислениям.

Все ключевые решения принимаются на уровне backend:

- кто пользователь;
- какой у него Telegram ID;
- какой у него email и телефон;
- кто его пригласил;
- какой тариф оплачен;
- до какой даты действует подписка;
- включено ли автопродление;
- какие Telegram-ресурсы ему доступны;
- какие партнёрские начисления созданы;
- какие выплаты можно проводить;
- кого нужно удалить из закрытых чатов.

Боты являются интерфейсами и исполнителями. Они не хранят бизнес-истину у себя локально.

## 2. Текущий состав внешних каналов

### 2.1. Активный пользовательский бот

В пользовательском контуре активным будет один бот: «Ассистент по обучению».

Его роль:

- точка входа из Telegram;
- переход в чат бесплатного обучения;
- первичная Telegram-идентификация пользователя;
- привязка Telegram ID к аккаунту на новой платформе;
- переход на сайт и в личный кабинет;
- показ тарифов;
- переход к оплате;
- показ доступных ресурсов;
- показ реферальной ссылки;
- базовые уведомления в Telegram.

Этот бот не является источником правды. Он обращается к центральному backend.

### 2.2. Новый бот для закрытых чатов

Для новых закрытых чатов создаётся новый технический бот доступа.

Рекомендованная модель:

- отдельный access-bot добавляется администратором во все новые закрытые Telegram-ресурсы;
- access-bot не обязательно активно общается с пользователями;
- access-bot выполняет команды backend: создать ссылку, проверить участника, одобрить заявку, удалить участника, зафиксировать ошибку;
- пользовательский бот «Ассистент по обучению» остаётся витриной и коммуникационным ботом;
- backend координирует оба бота.

Так чище архитектурно: один бот общается, второй молча делает грязную работу с чатами.

### 2.3. VK-сообщество

VK-сообщество является открытой площадкой и внешним каналом.

В MVP оно не участвует в:

- подписочной логике;
- закрытых доступах;
- оплатах;
- реферальной атрибуции;
- автоматическом удалении пользователей.

Ссылка на VK может отображаться на сайте и в личном кабинете как открытый ресурс.

### 2.4. Старые чаты

Старые чаты не мигрируются в новую подписочную систему.

Роль старых чатов:

- архивные материалы;
- старые договорённости с текущими участниками;
- новые подписочные правила к старым чатам не применяются;
- старые чаты не модерируются по новой подписке;
- старые чаты не являются новым платным продуктом.

Старые чаты можно показывать в личном кабинете как архивные ссылки, если это бизнесово уместно. Но они должны быть явно отделены от новых платных ресурсов.

Рекомендуемый статус для старых ресурсов в системе:

```text
ARCHIVE_VISIBLE
```

Это означает: ресурс можно показать пользователю, но backend не выдаёт и не снимает платный доступ к нему как к новому продукту. (дополнительно верифицировать)

## 3. Новые закрытые Telegram-ресурсы

Новые закрытые чаты и каналы будут созданы отдельно. Список ссылок по текущим материалам предоставлен; технические `chat_id` и точные типы ресурсов будут получены позже.

Именно эти новые ресурсы становятся управляемыми подписочными ресурсами.

Для каждого нового ресурса в системе нужно хранить:

- внутренний `resource_id`;
- название;
- тип: chat, group, supergroup, channel, forum_group;
- Telegram `chat_id`;
- публичную ссылку, если есть;
- служебную invite-ссылку, если используется;
- флаг `is_paid_resource`;
- флаг `is_active`;
- привязанный бот-администратор;
- список тарифов, которые дают доступ;
- политику входа: personal invite link или join request;
- показывать ли ресурс в личном кабинете;
- показывать ли ресурс в боте;
- порядок отображения.

Главное правило: в новые закрытые чаты попадают только пользователи, которые есть в базе платформы и имеют активный доступ. (продумать и подтвердить связь: открытый бот - бэк)

## 4. Модель доступа в Telegram

### 4.1. Базовая логика

У каждого пользователя есть аккаунт в центральной базе.

У пользователя есть Telegram-привязка.

У пользователя есть активная подписка, ручной доступ или промо-доступ.

На основе этого backend создаёт записи доступа к конкретным Telegram-ресурсам.

Доступ в Telegram не должен определяться по принципу «ну он там вроде сидит». Факт нахождения в чате — это следствие доступа, а не причина доступа. 

!!! Продумать оба пути для получения доступа: через бот и через сайт. В данный момент не описана регистрация.

### 4.2. Выдача доступа

Рекомендуемый путь для MVP:

1. Пользователь оплачивает тариф.
2. Backend создаёт подписку.
3. Backend рассчитывает список доступных Telegram-ресурсов.
4. Backend создаёт `access_entitlement` по каждому ресурсу.
5. Access-bot создаёт персональную invite-ссылку или ожидает join request.
6. Пользователь видит ссылки в личном кабинете и/или в боте.
7. Пользователь входит в нужные чаты.
8. Backend фиксирует факт входа, если Telegram присылает событие или если пользователь проверен через API.

### 4.3. Персональные ссылки и заявки

Для новых закрытых чатов лучше использовать один из двух вариантов.

Вариант A: персональная invite-ссылка.

- создаётся под конкретного пользователя;
- живёт ограниченное время;
- имеет лимит вступлений 1;
- хранится в базе;
- показывается в кабинете и боте;
- после использования больше не переиспользуется.

### 4.4. Синхронизация чатов с базой

Нужно реализовать два scheduler-процесса.

Первый scheduler: ежедневная проверка истекших подписок.

- запускается один раз в день;
- находит подписки и доступы, у которых истёк срок;
- проверяет, нет ли продления, автопродления, ручного доступа или промо-доступа;
- создаёт задачи на удаление из закрытых чатов;
- access-bot удаляет пользователя из соответствующих Telegram-ресурсов;
- backend пишет результат в лог.

Второй scheduler: еженедельная сверка Telegram-доступов.

- проходит по известным пользователям из базы;
- проверяет их статус в каждом ресурсе, где у них должен быть или не должен быть доступ;
- если пользователь есть в чате без права доступа, создаётся задача на удаление;
- если пользователь имеет право доступа, но не состоит в чате, ему можно заново показать ссылку;
- все расхождения фиксируются.

Важное ограничение: обычный Bot API не надо считать удобным инструментом для полной выгрузки всех участников чата. Поэтому правильная архитектура — контролировать вход с самого начала: новые чаты, персональные ссылки, join requests, отсутствие ручных ссылок и регулярная сверка известных пользователей. Проверка по кол-ву пользователей в чате и у сколько должен быть доступ.

### 4.5. Запрет ручного хаоса

Для новых закрытых чатов нужно организационное правило:

- не раздавать постоянные invite-ссылки;
- не добавлять пользователей вручную без записи в базе;
- не давать администраторам создавать сторонние ссылки без необходимости;
- все ручные доступы оформлять через админку;
- каждое ручное действие логировать.

Если человек оказался в закрытом чате без записи в базе, система должна считать это аномалией, а не «ну бывает». Вообще продумать пользовательский путь - решистрация через бота может быть быстрой (берем из него данные, создаем акк автоматом, просим ввести почту перед оплатой и тд если нужно) и чуть дольше с возможностью зарегаться через яндекс, вк, почту и по возможности вытаскивать данные для быстрой регистрации. а затем уже можно оплачивать. бизнес логику нужно продумать.

## 5. Тарифы

Тарифы в системе должны быть настраиваемыми сущностями, а не захардкоженными числами.

На текущий момент фиксируются три платных тарифа и бесплатный доступ.

### 5.1. Бесплатный доступ

Бесплатный доступ нужен для входа в экосистему.

В него входят:

- бот «Ассистент по обучению»;
- чат бесплатного обучения;
- открытые материалы;
- открытые ссылки, включая VK;

Бесплатный доступ не создаёт платную подписку и не создаёт партнёрское начисление.

### 5.2. Тариф «Тест-драйв»

Цена: 1800 ₽.

Ориентировочно: ≈ 20 $.

Партнёрское вознаграждение: 540 ₽.

Состав по текущим материалам:

- доступ в чат «Тест-драйв»;
- марафон по VK: настройка ВКонтакте и базовая автоматизация;
- доступ на онлайн-встречи клуба «Велес»;
- партнёрская программа клуба.

Не входит:

- библиотека знаний;
- канал «ИИ-Эксперт»;
- канал «Основы бизнеса онлайн».

### 5.3. Тариф «Старт в клубе»

Цена: 7800 ₽.

Ориентировочно: ≈ 87 $.

Партнёрское вознаграждение: 2340 ₽.

Состав по текущим материалам:

- чат «Тест-драйв»;
- основной рабочий чат «Велес»;
- актуальная информация по ИИ;
- юридические аспекты онлайн-бизнеса;
- финансовые инструменты для заработка;
- канал «Основы бизнеса онлайн»;
- пошаговое обучение по настройке Telegram и VK для бизнеса;
- создание автоматизации и воронок;
- практическое применение ИИ в бизнесе;
- партнёрская программа клуба.

Не входит:

- библиотека знаний;
- канал «ИИ-Эксперт».

### 5.4. Тариф «Максимум»

Цена: 14800 ₽.

Ориентировочно: ≈ 165 $.

Партнёрское вознаграждение: 4440 ₽.

Состав по текущим материалам:

- всё, что включено в «Тест-драйв»;
- всё, что включено в «Старт в клубе»;
- чат «Библиотека знаний»;
- все марафоны клуба: VK, Telegram, финансовое мышление, вирусные ролики и Reels, ИИ, внедрение ИИ в бизнес, нейроботы-продавцы, консультанты и техподдержка, CapCut;
- канал «ИИ-Эксперт»;
- стартовые знания по ИИ;
- пошаговое внедрение ИИ в свой бизнес для новичков.

### 5.5. Управление тарифами

В админке нужно уметь менять:

- название тарифа;
- цену;
- валюту;
- активность тарифа;
- состав доступных ресурсов;
- партнёрское вознаграждение;
- порядок отображения;
- описание для сайта;
- описание для личного кабинета.

Даже если сейчас тарифов всего три, код не должен жить так, будто человечество никогда не поменяет цену. Оно поменяет. Потом скажет, что «это же очевидно».

## 6. Срок подписки

Срок подписки: календарный месяц с момента оплаты.

Рабочее определение для системы:

```text
Дата начала = момент успешной оплаты.
Дата окончания = то же число следующего календарного месяца, в то же время.
```

Если в следующем месяце нет такого числа, дата окончания переносится на последний день следующего месяца.

Примеры:

- оплата 10 июня в 15:00 → доступ до 10 июля 15:00;
- оплата 31 января → доступ до 28 февраля или 29 февраля в високосный год;
- оплата 30 марта → доступ до 30 апреля.

Если пользователь продлевает подписку до окончания текущего периода, новый месяц добавляется к текущей дате окончания, а не обнуляет уже оплаченные дни.

Если подписка уже истекла, новая подписка начинается с момента новой успешной оплаты.

## 7. Продление, автопродление, отмена

### 7.1. Разовая оплата

Пользователь может оплатить один месяц без автопродления.

В этом случае:

- создаётся подписка на календарный месяц;
- автопродление выключено;
- перед окончанием подписки пользователь получает уведомление;
- если пользователь не продлил подписку, доступы снимаются после окончания периода.

### 7.2. Автопродление

Автопродление нужно закладывать сразу.

Полный цикл автопродления:

1. Пользователь выбирает тариф.
2. Пользователь соглашается на автопродление.
3. Платёжный провайдер сохраняет платёжный метод или создаёт регулярную подписку.
4. Backend хранит идентификатор сохранённого способа оплаты или подписки у провайдера.
5. В дату следующего списания backend инициирует или ожидает списание, в зависимости от провайдера.
6. При успешной оплате срок подписки продлевается.
7. При неуспешной оплате пользователь получает уведомление.
8. После финального неуспешного сценария автопродление отключается или переходит в статус ошибки.
9. Если доступ не продлён, пользователь удаляется из закрытых чатов после окончания оплаченного периода.

### 7.3. Отключение автопродления

Пользователь должен иметь возможность отключить автопродление:

- в личном кабинете;
- возможно, через кнопку в Telegram-боте;
- возможно, через ссылку провайдера, если провайдер это требует.

Отключение автопродления не отменяет уже оплаченный доступ.

Если пользователь оплатил до 10 июня и отключил автопродление 20 июня, доступ остаётся до 10 июля.

### 7.4. Неуспешное автосписание

Рекомендуемый сценарий:

- первая ошибка списания: уведомить пользователя;
- повторная попытка через 1 день;
- вторая ошибка: уведомить повторно;
- финальная попытка в день окончания подписки или на следующий день;
- если списание не прошло, подписка не продлевается;
- доступ снимается после окончания оплаченного периода.

Количество попыток должно быть настраиваемым.

## 8. Downgrade и upgrade

### 8.1. Downgrade

Downgrade — переход на более низкий тариф.

Бизнес-правило зафиксировано:

- текущий оплаченный тариф действует до конца оплаченного периода;
- деньги за текущий период не возвращаются;
- пониженный тариф начинает действовать со следующего периода;
- пользователь заранее выбирает, какой тариф будет следующим;
- если включено автопродление, следующее списание должно пройти уже по новому тарифу;
- если автопродление выключено, пользователю просто показывается выбранный следующий тариф для оплаты.

### 8.2. Что такое upgrade

Upgrade — это переход на более высокий тариф до окончания текущего оплаченного периода.

Это не вопрос «какие у нас тарифы», они уже есть. Это вопрос «как считать деньги и когда выдавать доступ», чтобы потом не городить арифметику на коленке, как будто калькулятор — это предмет роскоши.

### 8.3. Рекомендуемое правило upgrade для MVP

Рекомендуемое правило:

- upgrade применяется сразу после успешной доплаты;
- доступы старшего тарифа выдаются сразу;
- дата окончания подписки обновляется от текущей даты;
- сумма доплаты считается как разница между текущим тарифом по оставшимся дням и новым тарифом .;
- включить сразу пропорциональный расчёт по оставшимся дням.

Для MVP лучше использовать простую модель:

```text
Доплата = цена нового тарифа - цена текущего тарифа по оставшимся дням.
```


При upgrade партнёрское начисление не создаётся повторно, потому что партнёрка платится только с первой платной оплаты.

Архитектурно нужно хранить upgrade как отдельный order/payment type, чтобы потом можно было поменять формулу без переписывания подписок.

## 9. Платежи

### 9.1. Общая схема оплаты

Оплата должна проходить через универсальную платёжную прослойку.

Пользователь может начать оплату:

- из Telegram-бота;
- из личного кабинета;
- с сайта/лендинга;
- по прямой ссылке на checkout.

Базовый поток:

1. Пользователь выбирает тариф.
2. Frontend или бот отправляет запрос в backend.
3. Backend создаёт order.
4. Backend создаёт payment через активный payment provider adapter.
5. Провайдер возвращает платёжную ссылку или параметры виджета.
6. Пользователь оплачивает на странице провайдера или в виджете.
7. Провайдер отправляет webhook в backend.
8. Backend проверяет подпись и статус.
9. Backend активирует или продлевает подписку.
10. Backend создаёт доступы к Telegram-ресурсам.
11. Backend создаёт партнёрское начисление, если это первая платная оплата и есть подтверждённый реферер.

### 9.2. В MVP сначала делаем mock provider

До выбора реального провайдера нужно реализовать `MockPaymentProviderAdapter`.

Он должен имитировать:

- создание платежа;
- успешную оплату;
- неуспешную оплату;
- возврат;
- автопродление;
- webhook;
- повтор webhook;
- задержку оплаты;
- ошибку провайдера.

Это позволит спокойно построить подписки, доступы, партнёрку, админку и scheduler без ожидания договора с платёжкой. 

### 9.3. Интерфейс платёжного адаптера

Нужно заложить интерфейс примерно такого уровня:

```java
interface PaymentProviderAdapter {
    CreatePaymentResult createPayment(CreatePaymentRequest request);
    PaymentStatusResult getPaymentStatus(String providerPaymentId);
    RefundResult refund(RefundRequest request);
    CreateRecurringResult createRecurring(CreateRecurringRequest request);
    CancelRecurringResult cancelRecurring(CancelRecurringRequest request);
    WebhookParseResult parseWebhook(HttpRequest request);
    ProviderCapabilities getCapabilities();
}
```

Платформа не должна знать внутренние тонкости ЮKassa, CloudPayments, Т-Банка или другого провайдера. Платформа знает только свои сущности: order, payment, subscription, refund, recurring agreement, webhook event.

### 9.4. Минимальные сущности оплаты

Нужны сущности:

- `payment_provider`;
- `payment_provider_config`;
- `order`;
- `payment`;
- `payment_attempt`;
- `payment_webhook_event`;
- `refund`;
- `recurring_agreement`;
- `recurring_charge_attempt`;
- `invoice_link` или `checkout_session`.

### 9.5. Статусы платежа

Минимальный набор статусов:

```text
CREATED
WAITING_CONFIRMATION
PENDING
SUCCEEDED
FAILED
CANCELLED
REFUND_PENDING
REFUNDED
PARTIALLY_REFUNDED
CHARGEBACK
MANUAL_CONFIRMED
```

`MANUAL_CONFIRMED` нужен для будущих нестандартных способов оплаты, например крипты или ручных переводов, но в MVP лучше не начинать с них.

### 9.6. Провайдеры для изучения

Провайдера сейчас не выбираем. Сначала делаем универсальную прослойку и mock provider.

Кандидаты для изучения:

1. ЮKassa.
   Подходит как базовый кандидат для рублёвых платежей, API, виджета, платёжной формы, чеков и автоплатежей. В документации ЮKassa автоплатежи описаны как сохранение способа оплаты и последующее списание по API; также явно указано, что пользователь должен согласиться с условиями автоплатежей и иметь возможность отключения.

2. CloudPayments.
   Подходит как кандидат для виджета, подписок, рекуррентных платежей и гибкой checkout-логики. В документации есть отдельные разделы по рекуррентным платежам, запуску/остановке регулярных платежей, графику платежей и смене суммы.

3. Т-Банк.
   Подходит как кандидат для интернет-эквайринга, платёжной формы, оплаты по ссылке, рекуррентных платежей, T-Pay, СБП и массовых выплат. В документации Т-Банка указана возможность подключить рекуррентные платежи, когда покупатель соглашается на регулярные списания, а следующие оплаты проходят без повторного ввода данных.

4. Robokassa.
   Подходит как кандидат для простой интеграции через API, платёжные формы, уведомления ResultURL/SuccessURL/FailURL, тестовый режим, онлайн-кассу и поддержку карт/СБП/электронных способов оплаты.

5. PayKeeper.
   Подходит как кандидат для интернет-эквайринга, СБП, онлайн-кассы, рекуррентных платежей, привязки карты, массовых выплат и API/виджетов.

6. Prodamus.
   Нужно отдельно изучить как возможный сервис для инфобизнеса, платёжных страниц и подписок. Не выбираем без проверки API, рекуррентных платежей, чеков, webhooks и условий.

### 9.7. Критерии выбора провайдера

Сравнивать нужно не по принципу «где кнопка красивее», хотя, конечно, мир пал и это тоже влияет.

Критерии:

- поддержка рекуррентных платежей;
- возможность разовой оплаты;
- понятная отмена автопродления;
- webhooks;
- тестовый режим;
- качество документации;
- Java-friendly API или нормальный REST;
- платёжная страница или виджет;
- поддержка чеков/онлайн-кассы;
- возвраты;
- комиссии;
- сроки вывода денег;
- массовые выплаты, если хотим автоматизировать партнёров;
- поддержка СБП;
- условия подключения для ИП/ООО;
- риск блокировок/ограничений;
- возможность передавать metadata: user_id, order_id, tariff_id, referral_relation_id.

## 10. Аккаунт и авторизация

### 10.1. Центральный аккаунт

Главный аккаунт — внутренний аккаунт платформы.

Telegram ID, email и телефон являются привязками к аккаунту.

Модель:

```text
user
user_identity
telegram_account
email_contact
phone_contact
```

Telegram обязателен для доступа в закрытые Telegram-ресурсы.

Email нужен для:

- чеков;
- уведомлений;
- восстановления доступа;
- личного кабинета;
- будущих рассылок.

Телефон нужен как дополнительный канал связи и идентификатор, но степень обязательности можно настроить.

### 10.2. Регистрация через Telegram

Сценарий:

1. Пользователь запускает бота «Ассистент по обучению».
2. Бот отправляет Telegram ID, username, first_name, last_name в backend.
3. Backend создаёт или обновляет аккаунт.
4. Backend проверяет реферальный параметр, если пользователь пришёл по deep link.
5. Backend возвращает пользователю ссылку в личный кабинет.
6. Ссылка содержит одноразовый login token.
7. Пользователь попадает в кабинет уже с привязанным Telegram.
8. В кабинете пользователь добавляет email и телефон.

### 10.3. Регистрация через сайт

Сценарий:

1. Пользователь заходит на сайт.
2. Создаёт аккаунт через email или начинает checkout.
3. Перед получением доступа к Telegram ему нужно привязать Telegram.
4. Сайт показывает кнопку «Привязать Telegram».
5. Кнопка ведёт в бота с одноразовым deep link token.
6. Бот отправляет token в backend.
7. Backend связывает Telegram ID с аккаунтом.

### 10.4. Оплата без Telegram

Технически можно разрешить оплату до привязки Telegram, но доступ к Telegram-чатам нельзя выдать без Telegram ID.

Рекомендуемое правило:

- оплатить можно после email;
- для получения ссылок в закрытые Telegram-ресурсы Telegram должен быть привязан;
- после оплаты пользователь видит задачу «Привяжите Telegram, чтобы получить доступ».

## 12. Реферальная программа

### 12.1. Базовое правило

Реферальная программа один уровень.

Платим только прямому пригласителю.

Платим только с первой платной оплаты пользователя.

Не платим:

- с продлений;
- с автопродлений;
- с upgrade;
- с промо-доступов;
- с бесплатного доступа.

Дерево приглашений можно хранить глубже для аналитики, но финансовая выплата только один уровень.

### 12.2. Принцип атрибуции

Принцип атрибуции:

```text
Первый подтверждённый реферер выигрывает.
```

Если пользователь впервые пришёл по ссылке партнёра и реферер был подтверждён, он закрепляется за пользователем.

Последующие переходы по другим ссылкам автоматически реферера не меняют.

Смена реферера возможна только вручную через админку, с причиной и логом.

### 12.3. Реферальные ссылки

Нужен единый центральный реферальный код.

Он должен работать:

- на сайте;
- в Telegram-боте;
- в checkout;
- в личном кабинете;
- в будущих каналах.

Код генерирует центральный backend, а не бот.

### 12.4. Партнёрские суммы

На текущий момент фиксируются такие суммы:

```text
Тест-драйв, 1800 ₽ → партнёру 540 ₽
Старт в клубе, 7800 ₽ → партнёру 2340 ₽
Максимум, 14800 ₽ → партнёру 4440 ₽
```

В системе это должно храниться как настраиваемое правило.

Нужно поддержать два варианта расчёта:

- фиксированная сумма по тарифу;
- процент от оплаты.

Для текущего запуска можно использовать фиксированные суммы, которые совпадают с 30%.

### 12.5. Статусы партнёрских начислений

Минимальный набор статусов:

```text
PENDING_HOLD
APPROVED
PAYOUT_SCHEDULED
PAID
CANCELLED
REVERSED
DISPUTED
```

### 12.6. Hold перед выплатой

Hold перед выплатой: 15 дней.

Комиссия создаётся после первой успешной оплаты, но сначала попадает в `PENDING_HOLD`.

Через 15 дней, если нет возврата, chargeback, ручной блокировки или спора, комиссия переходит в `APPROVED`.

### 12.7. График выплат

График выплат должен быть настраиваемым.

Рекомендуемое начальное правило:

- расчёт выплат один раз в месяц;
- в выплату попадают только комиссии в статусе `APPROVED`;
- комиссия должна пройти 15-дневный hold;
- выплаты суммируются если их в APPROVED больше одной
- при необходимости можно переключить график на два раза в месяц без изменения модели данных.

В системе нужно хранить `payout_batch`, чтобы каждая выплата была отдельным реестром.

### 12.8. Возвраты и партнёрка

Правила:

Если возврат произошёл во время hold:

- комиссия переводится в `CANCELLED`;
- в выплату не попадает
- должно отображаться в ui

Если возврат произошёл после `APPROVED` (такого технически не должно быть), но до выплаты:

- комиссия переводится в `REVERSED` или `CANCELLED`;
- в выплату не попадает;
- причина сохраняется.

Если возврат произошёл после выплаты партнёру:

- создаётся отрицательная корректировка;
- сумма удерживается из будущих выплат партнёра;
- если будущих выплат нет, это остаётся как отрицательный баланс или ручной финансовый вопрос
- учесть ситуацию при разработке, сделать невозможной

Если chargeback:

- комиссия блокируется;
- партнёр может получить флаг риска;
- дальнейшие выплаты по спорному пользователю не проводятся до решения.

### 12.9. Антифрод рефералки

Нужно запретить:

- self-referral по Telegram ID;
- self-referral по email;
- self-referral по телефону;
- повторное начисление за одного и того же пользователя;
- начисление с бесплатного доступа;
- начисление с ручного админского доступа;
- начисление после возврата;
- автоматическую смену реферера последним кликом.

Нужно хранить:

- откуда пришёл пользователь;
- какой ref code использован;
- когда реферер закреплён;
- кто изменил реферера вручную;
- почему изменил;
- какие начисления созданы;
- какие начисления отменены;
- какие выплаты проведены.

## 13. Промокоды

Промокоды пока под вопросом, но архитектурно их лучше заложить как опциональную сущность.

В MVP не выводим сложный UI промокодов, но таблицы и модель лучше предусмотреть.

Типы промокодов:

- бесплатный доступ на срок;
- скидка;
- тестовый период;
- ручной промо-доступ.

Правило для партнёрки:

- если промокод даёт 100% скидку, партнёрское начисление не создаётся;
- если промокод даёт частичную скидку, партнёрское начисление считается по отдельному правилу, которое нужно явно задать;
- по умолчанию для MVP лучше не начислять партнёрку с промокодных оплат, пока бизнес не зафиксирует правило.

## 18. Scheduler-и

Нужны scheduler-и:

1. `subscription-expiration-scheduler` — ежедневная проверка истекших подписок.
2. `telegram-access-revoke-scheduler` — удаление из закрытых чатов.
3. `telegram-access-reconciliation-scheduler` — еженедельная сверка базы и Telegram.
4. `renewal-notification-scheduler` — уведомления об окончании подписки.
5. `recurring-payment-scheduler` — автопродления, если провайдер требует инициировать списание с нашей стороны.
6. `partner-commission-approval-scheduler` — перевод комиссий из hold в approved после 15 дней.
7. `payout-batch-scheduler` — формирование выплатных реестров.
8. `outbox-retry-scheduler` — повтор неуспешных интеграционных событий.

Все scheduler-и должны быть безопасны к повторному запуску.

Для этого нужны:

- idempotency key;
- scheduler lock;
- retry count;
- статус задачи;
- лог ошибки;
- dead-letter для задач, которые не удалось выполнить.

## 21. Что больше не считаем открытым вопросом

Следующие решения считаются зафиксированными:

- активный пользовательский бот один: «Ассистент по обучению»;
- для новых закрытых чатов создаётся новый бот доступа;
- старые чаты не мигрируются;
- старые чаты можно показывать как архивные ресурсы;
- основной продукт строится вокруг новых закрытых Telegram-ресурсов;
- Telegram-доступ строится от базы данных;
- центральный backend — источник правды;
- тарифов три плюс бесплатный доступ;
- цены: 1800 ₽, 7800 ₽, 14800 ₽;
- партнёрские выплаты: 540 ₽, 2340 ₽, 4440 ₽;
- партнёрка один уровень;
- партнёрка только с первой платной оплаты;
- первый подтверждённый реферер выигрывает;
- hold перед выплатой 15 дней;
- автопродление нужно делать сразу;
- личный кабинет нужен полноценный;
- платёжный провайдер пока не выбран;
- сначала делаем универсальную платёжную прослойку и mock provider.

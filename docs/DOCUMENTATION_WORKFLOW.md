# Правила работы с документацией

## Назначение

Этот файл фиксирует правила работы с документацией проекта `veles-platform`.

Цель — не терять решения владельца, не смешивать бизнес-логику с архитектурой и не переписывать базовые документы без подтверждения.

## Типы документов

### Бизнес-логика

Фиксирует продуктовые и организационные правила:

- тарифы;
- подписки;
- оплаты;
- автопродление;
- реферальную программу;
- партнёрские выплаты;
- правила доступа;
- пользовательские сценарии;
- модерацию;
- уведомления;
- личный кабинет;
- бизнес-функции Telegram-бота.

Бизнес-логика пишется для владельца проекта и должна быть понятна без чтения кода.

### Архитектура

Фиксирует техническую реализацию:

- backend;
- Telegram Bot API;
- платёжные адаптеры;
- базу данных;
- scheduler-и;
- интеграции;
- безопасность;
- роли;
- инфраструктуру;
- API;
- миграции.

Архитектуру нельзя смешивать с бизнес-логикой. Если вопрос технический, он должен идти в архитектурный документ или ADR.

### Открытые вопросы

Открытые вопросы — короткий рабочий список для владельца проекта.

Формат вопроса:

- вопрос;
- почему нужен ответ;
- текущая гипотеза, если она есть;
- куда перенести ответ после решения.

После ответа вопрос удаляется из файла открытых вопросов, а решение переносится в бизнес-логику или архитектуру.

### Внутренние заметки

Внутренние заметки нужны для сохранения контекста длинных обсуждений.

Они могут быть подробнее, чем открытые вопросы, но не являются финальным источником правды.

### ADR

ADR используются для технических архитектурных решений, когда есть несколько вариантов и нужно зафиксировать выбор.

ADR пишутся отдельно от бизнес-логики.

## Правила изменения базовых документов

Базовые документы бизнес-логики и архитектуры нельзя менять без approval владельца.

Рабочий порядок:

1. Собрать вводные.
2. Разделить вводные на бизнес-логику, архитектуру, открытые вопросы и заметки.
3. Подготовить короткое summary для владельца.
4. Получить approval.
5. Только после этого вносить изменения в базовые документы.
6. После внесения убрать закрытые вопросы из файла открытых вопросов.

## Запреты

Нельзя:

- переписывать документ с нуля, если задача — уточнить текущий;
- смешивать бизнес-решения и техническую реализацию в одном разделе;
- превращать открытые вопросы в длинный аналитический документ;
- фиксировать предположение как решение;
- удалять ранее зафиксированное решение без отдельного подтверждения;
- вносить правки в базовую бизнес-логику или архитектуру без approval.

## Рекомендуемая структура репозитория

```text
docs/
  business/
    BUSINESS_LOGIC.md
  architecture/
    ARCHITECTURE.md
    adr/
  open-questions/
    BUSINESS_OPEN_QUESTIONS.md
    ARCHITECTURE_OPEN_QUESTIONS.md
  notes/
```

## Статусы вопросов

```text
OPEN — вопрос открыт.
PROPOSED — есть предложенное решение, нужен approval.
APPROVED — решение подтверждено, нужно перенести в базовый документ.
MOVED — решение перенесено в базовый документ, вопрос можно убрать.
BLOCKED — нужен внешний ввод: юрист, платёжный провайдер, владелец Telegram-ресурса.
```

## Текущий рабочий принцип

Сначала закрывается бизнес-логика. Затем отдельно прорабатывается архитектура.

Внешний сайт и лендинг не входят в текущий контур. Текущий контур — личный кабинет, аккаунт, Telegram-бот, подписки, оплаты, доступы, партнёрка и админские процессы.
